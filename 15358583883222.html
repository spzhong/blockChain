    整个交易的过程中，其实最难理解是索引和输入输出，接下来我分别介绍一下

    • 输出：通常理解为锁定脚本，决定谁可以花费这币交易
    • 输入：通常理解为解锁脚本，证明我拥这笔交易的所有权
    • UTXO：未花费交易，获取输出脚本为当前钱包地址的所有交易信息列表，然后加上索引，匹配所有交易的输入是否被引用，否的话称为“UTXO”
    • 索引：在一个交易中，可能会被分解为多个UTXO,就是通过索引来确定该交易属于那个钱包地址的UTXO

    输入和输出，是那个先产生的呢？严格来说先产生输出，因为可以把创造新别特币的交易理解为没有输入，但是可以生产出输出，所以我们理解的交易也是现从输出开始的

    2.首先要知道钱包是如何创建的

    1.私钥                  随机32位字符（256位随机数）
2.公钥                  椭圆曲线加密SECP2561算法
3.公钥哈希值           SHA256(公钥哈希)
4.公钥哈希值              Ripemd160(公钥哈希值)
5.校验码                SHA256(SHA256(1个字节版本号0x00+公钥哈希值))，取前4个字节
6.钱包地址              base58(1个字节版本号0x00+公钥哈希值+校验码)
例如:1LYBoAY8u4sCrNEpVoB3SxkoKDm8aAF5yz

    3.了解比特币涉及到的块，交易的数据结构定义

    #块的结构
type Block struct {
        hash               string       哈希
        confirmations      string       确认次数
        size               uint64       块的大小
        height             uint64       块的高度
        version            uint32       块的版本
        merkleroot         string       默克尔树（后期讲解）
        tx                 []           交易的详细信息集合
        time               uint32       块的时间
        nonce              uint32       随机数（最大2^32，后期讲解）
        bits               string       当前目标hash值
        difficulty         float64      难度系数（用于挖矿控制，后期讲解）
        previousblockhash  string       上一个区块的hash
}

#交易的结构

    挖矿所得的交易实际上并不是一个交易，coinbase交易需要经过后续100块（10min*100=大约需要17小时）确认，方能真正的使用该币交易，防止出现多个矿工同时挖出块，最终会以最长块的为准；

    coinbase

    该字段由挖出此Block的人填写，这是一种“特权”;

    coinbase字段可以是最小2字节，最大100字节的任意字符

    sequence

    交易的序列，该值的范围是0-232 -1，该值往往是和下面的locktime配合使用的，当sequence=232 -1，则就忽略locktime的功能；
    当locktime是未来的时间，且sequence<232 -1的时候，那么到了loacktime的时候，以sequence最大的准，其它的交易将会从交易池中移除

    这个字段的作用主要是为了解决提前解锁未来交易，通常的做法是再发起一个相同的交易，将sequence设为232 -1，矿工就会忽略sequence为其它的交易，将sequence=232 -1（表示最终确认的交易）打包写入块中

    locktime

    在将来的脚本，智能合约中运用的比较多，后续在脚本语言详细的介绍

    locktime 定义
    0 该交易立即生效，可以矿工打包
    < 5亿 未来block的高度，当小于该block高度之前处于锁定状态
    >= 5亿 未来Unix时间戳，当小于该Unix时间戳之前处于锁定状态

    6.获取自己的钱包余额

    当有了上面的两个交易，那么我们的钱包中就拥有了比特币，接下来怎么计算自己的钱包余额呢？
    首先确保你的节点是一个全节点（包含全部的区块数据）

    遍历所有区块数据

    获取指定钱包地址的交易输出：匹配交易中scriptPubKey中的addresses字段，得到该交易的id，交易序号，及交易金额，将其存放到HashMap中(交易的id+交易序号作为key)

    优化建议思考：
    1.钱包地址上增加创建的时间戳，只需遍历区块时间大于该钱包上时间戳即可
    2.在遍历区块的时候，将该钱包最早的产生交易的块高度进行缓存，下次遍历大于该缓存高度的区块即可
    3.运用分布式的区块遍历查询
    4.如何快速遍历区块的数据也是一门学问，例如内存问题，多线程查询等等

    得到未花费的交易输出

    遍历得到的HashMap，根据交易的id和交易序号，查询区块中是否引用了该交易，执行的方法就是交易的id和交易序号是否在其他交易的输入中，未被引用的交易输出就是未被花费的交易输出（简称UTXO）

    优化建议思考：
    1.存储区块交易的时候，可将交易的id+交易序号作为区块中交易HashKey
    2.遍历大于最早产生交易的块

    得到钱包中余额

    累加UTXO中的所有交易金额，就是该钱包的余额

    7.那么如何花费自己的UTXO呢

    1. 首先获得自己所有的UTXO，请参考获取自己的钱包余额

    2. 转账到指定钱包（钱包地址，以及转账金额）

    3. 遍历UTXO得到交易的输出，需要通过自己的私钥签名验证，才能正真的花费该笔交易（这一步具体的会在脚本中讲解）

    4. 将验证通过的多笔交易输出及签名组装成新的交易输入，具体的数据结构请参考普通转账的交易

    5. 累加交易输出的交易金额>=转账金额进行break

    6. 组装交易的输出，如果有可能需要找零，再多加一个输出指定到自己钱包中，具体的数据结构请参考普通转账的交易

      每笔交易都需要交易费，即是给矿工打包点小费，交易费=输入的总和-输出的总和;
      交易费是按照交易的字节长度而变化，大约10个satoshi/byte（目前增加了），一笔常规的交易大约两百多个字节，那么大约普通的一笔交易费大约2000个satoshi(0.00002BTC)

      如果忘记了给自己构造交易中找零的输出，那么系统会将（输入的总和-输出的总和）作为交易费，实际意思就是“不用找了”

    7. 得到完整的交易信息，在进行交易的验证

    8. 进行全网的广播，每个节点都可以独立的进行验证该交易

    9. 由矿工验证通过，打包进入交易池中

    10. 挖矿产生新的区块写入链中，然后进行全网的广播

    11. 其他节点进行同步所有的区块

    12. 理论上，该笔交易已经完成了，但是为了防止多个矿工挖出区块导致短暂的分叉，所有的交易需经过6个块的（大约需要1个小时）确认，方为认为交易真正的有效；

    8.交易的验证

    由于比特币的源码迭代，不同版本的交易验证多少有些出入，下面仅作为参考

    • 交易的语法和数据结构必须正确。
    • 输入与输出列表都不能为空。
    • 交易的字节大小是小于MAX_BLOCK_SIZE的。
    • 每一个输出值，以及总量，必须在规定值的范围内 （小于2,100万个币，大于0）。
    • 没有哈希等于0，N等于-1的输入（coinbase交易不应当被中继）。
    • nLockTime是小于或等于INT_MAX的。
    • 交易的字节大小是大于或等于100的。
    • 交易中的签名数量应小于签名操作数量上限。
    • 解锁脚本（scriptSig）只能够将数字压入栈中，并且锁定脚本（scriptPubkey）必须要符合isStandard的格式 （该格式将会拒绝非标准交易）。
    • 池中或位于主分支区块中的一个匹配交易必须是存在的。
    • 对于每一个输入，如果引用的输出存在于池中任何的交易，该交易将被拒绝。
    • 对于每一个输入，在主分支和交易池中寻找引用的输出交易。如果输出交易缺少任何一个输入，该交易将成为一个孤立的交易。如果与其匹配的交易还没有出现在池中，那么将被加入到孤立交易池中。
    • 对于每一个输入，如果引用的输出交易是一个coinbase输出，该输入必须至少获得COINBASE_MATURITY (100)个确认。
    • 对于每一个输入，引用的输出是必须存在的，并且没有被花费。
    • 使用引用的输出交易获得输入值，并检查每一个输入值和总值是否在规定值的范围内 （小于2100万个币，大于0）。
    • 如果输入值的总和小于输出值的总和，交易将被中止。
    • 如果交易费用太低以至于无法进入一个空的区块，交易将被拒绝。
    • 每一个输入的解锁脚本必须依据相应输出的锁定脚本来验证。

    9.交易的脚本初窥

    什么是脚本语言

    是一种类似Forth的逆波兰表达式的基于堆栈的执行语言，在比特币交易中的输出和输入都是基于堆栈的执行的语言编写的。当一笔交易在验证的时候，每一个输入值的解锁脚本与对应的输出锁定脚本进行匹配执行

    脚本语言有哪些特点

    编程语言简单，可在硬件上执行，安全

    • 图灵非完备性：包含了许多的操作码，但故意限定为一种模式除了有条件的流控制以外，美哟循环或复杂的控制能力
    • 去中心化验证：脚本语言没有中心主体，自然也不会有中心主体会在脚本被执行后对其进行保存（与其对应的是runtime动态编程的语言），所以一个脚本在任何系统上都以相同的方式执行

    一个简单的脚本

    例如脚本：2 3 OP_ADD 5 OP_EQUAL
    执行的顺序是从左至右依次进行入栈

    栈 脚本 说明

    3 2 3 OP_ADD 5 OP_EQUAL 3入栈，且有2个元素

    栈顶为5，且有1个元素
    5 2 3 OP_ADD 5 OP_EQUAL 5入栈，且有2个元素
    5

    10.交易的数字签名

    什么是数字签名

    • 数字签名的定义

      维基百科：数字签名是用于证明数组消息或文档的真实性的数学方案，有效的数字签名给一个容易接受的理由去相信；

      1.消息是由已知的发送者创建(身份认证性)
      2.发送者不可否认已发送的消息(不可否认性)
      3.消息在传输中未被更改(数据完整性)

    • 比特币使用的数字签名算法是椭圆曲线数字签名算法（ECDSA）,是用于基于椭圆曲线私钥/公钥对的数字签名的算法

    • 数字签名在脚本语言中运用，脚本函数如下，这些脚本函数会在后续的脚本中详细说明

      OP_CHECKSIG
      OP_CHECKSIGVERIFY
      OP_CHECKMULLSIG
      OP_CHECKMULTISGVERIFY

    • 数字签名的用处
      签名证明私钥的所有者，授权证明是不可否认的，签名证明交易在签字之后没有也不可能被任何人修改

    数字签名是如何工作的

    • 创建数字签名

      Sig = F{sig}(F{hash}(m),dA)

      dA是签名私钥
      m是交易（或其部分）
      F{hash}是散列函数
      F{sig}是签名算法
      Sig是结果签名

      函数F{sig}产生由两个值组成的签名sig，通常称为R(公钥)和S(签名)，Sig结果 = (R,S)
      这只是一个简单的数字签名，实际的算法过程较为复杂，想了解更多的同学请Google

    • 签名序列化(DER)

      3045022100884d142d86652a3f47ba4746ec719bbfbd040a570b1deccbb6498c75c4ae24cb022
      04b9f039ff08df09cbe9f6addac960298cad530a863ea8f53982c09db8f6e381301

      0x30:表示DER序列的开始
      0x45:序列的长度（69个字节）
      0x02:一个整数值
      0x21:整数的长度（33个字节）
      R:00884d142d86652a3f47ba4746ec719bbfbd040a570b1deccbb6498c75c4ae24cb
      0x02:一个整数值
      0x20:整数的长度（32个字节）
      S签名:04b9f039ff08df09cbe9f6addac960298cad530a863ea8f53982c09db8f6e3813
      0x01:后缀表示使用的哈希的类型

    • 如何验证签名

      验证签名，必须有签名(R和S)，序列化交易和公钥（对应于创建签名的私钥），本质上签名的验证意味着“只有生成此公钥的私钥的所有者，才能在此交易上产生此签名”

      签名验证算法采用消息（交易或其部分的哈希值），签名者的公钥和签名(R和S值)，如果签名对该消息和公钥有效，则返回TRUE值

      具体的验证方法参考ECDSA数学

    数字签名哈希（SIGHASH）

    • 有三个SIGHASH标志：ALL,NONE和SINGLE

      SIGHASH FLAG 值 描述
      ALL 0x01 签名用于所有的输入和输出
      NONE 0x02 签名用于所有的输入，但是不包含输出
      SINGLE 0x03 签名应用于所有的输入，但只有一个输入作为相同的索引数的输出

    • 修饰符ANYONECANPAY与SIGHASH进行组合，通过OR运算

      SIGHASH FLAG 值 描述
      ANYONECANPAY 0x80 只有一个输入被签名，其余的（及序列号）打以进行修改
      ALL|ANYONECANPAY 0x81 签名用于所有的输入和输出
      NONE|ANYONECANPAY 0x82 签名用于所有的输入，但是不包含输出
      SINGLE|ANYONECANPAY 0x83 签名应用于一个输入，并且输出作为相同的索引

    • SIGHASH标示在签名和验证期间应用的方式是建立交易的副本和删节其中的某些字段（设置长度为零并清空）

      ALL | ANYONECANPAY

      可以做“众筹”交易，试图筹集资金的人可以用单笔输出来构建一个交易，单笔输出将“目标”金额付给发起人。这样的交易显然是无效的，因为没有输入，但是其他人可以通过添加自己的输入作为捐赠来修改他们，他们用ALL | ANYONECANPAY签署自己的输入，除非收集到足够的输入以达到输出的价值，否则交易无效，每次捐赠都是一项“抵押”，直到募集整个目标金额

      NONE

      该结构用于创建特定数量的“不记名支票”或“空白支票”，它对输入进行承若，但允许输出锁定脚本被更改。任何人都可以将自己的比特币地址写入输出锁定脚本并兑换交易。然而，输出值本身被签名锁定。

      NONE | ANYONECANPAY

      这种构造可以用来建造一个“吸尘器”，在他们的钱包中拥有微小的UTXO的用户无法花费的这些费用，因为手续费超过了这些微小的价值。借助这种类类型的签名，微小UTXO可以为任何人捐赠，以便随时的收集和消费

    ECDSA数学

    • 公钥和私钥

      公钥的主要作用：加密，验证签名
      私钥的主要作用：签名，解密

      特性：通过私钥可以计算出公钥，反之则不行

      公钥加密：公钥加密的内容可以用私钥解密--只有私钥持有者才能解密，可逆向的到明文

      私钥签名：私钥签名的内容可以用公钥验证--公钥能验证的签名均视为私钥持有人所签署

    • 椭圆曲线数字签名算法(ECDSA)

      用户密钥对：pubkey,privkey
      待签名的信息：M
      签名：Signature(M) = (r,s)

      签名过程：
1、根据ECC算法随机生成一个密钥对(k, R), R=(xR, yR)
      
2、令 r = xR mod n，如果r = 0，则返回步骤1
      
3、计算 H = Hash(M)
      
4、按照数据类型转换规则，将H转化为一个big endian的整数e
      
5、s = k^-1 (e + rd) mod n，若s = 0, 则返回步骤1
      
6、输出的S =(r,s)即为签名

      验证过程：
1、计算 H = Hash(M)
      
2、按照数据类型转换规则，将H转化为一个big endian的整数e
      
3、计算 u1 = es^-1 mod n, u2 = rs^-1 mod n
      
4、计算 R = (xR, yR) = u1G + u2Q, 如果R = 零点，则验证该签名无效
      
5、令 v = xR mod n
      
6、若 v == r，则签名有效，若 v ≠ r, 则签名无效

      备注：具体参考比特币系统的公钥密码学方案和ECDSA签名算法介绍

    随机性在签名中的重要性

    在ECDSA Math中，签名生成算法使用随机密钥k作为临时私钥-公钥对的基础。k值不重要，只要它是随机的；如果使用相同的值k在不同的消息（交易）上产生两个签名，那么签名私钥可以由任何人计算，这样会导致私钥的泄漏。

    比特币中签署的交易算法，则必须使用RFC6979或类似的确定性随机算法来确保为每个交易生成不同的k值
